## Methodology

### Setup

Each engineering prompt is run twice under identical conditions:

- **OpenEng (skills-loaded):** Claude with all OpenEng skills loaded via the skills library. The relevant domain skills are activated based on the prompt content.
- **Base Claude (no skills):** Claude with no OpenEng skills loaded -- relying only on base training knowledge.

Both runs use the same Claude model version and temperature settings. The only variable is skill availability.

### Scoring Criteria

A human engineer scores both outputs on 4 criteria using a 1-5 scale:

| Score | Meaning |
|-------|---------|
| 1 | Poor -- fundamentally incorrect or missing key elements |
| 2 | Below average -- major gaps in approach or accuracy |
| 3 | Adequate -- generally correct but lacks depth or specificity |
| 4 | Good -- technically sound with appropriate detail |
| 5 | Excellent -- comprehensive, accurate, and directly applicable |

**Criteria:**

1. **Technical Accuracy** -- Are calculations, methods, and technical recommendations correct?
2. **Completeness** -- Does the response cover all relevant steps, checks, and considerations?
3. **Uganda/East Africa Relevance** -- Does the response reference appropriate local standards (MoWT, UNRA, NEMA), use locally relevant parameters, and account for East African conditions?
4. **Standard Compliance** -- Does the response cite correct manual/code references and follow proper procedures per those standards?

### Blinding Protocol

> **Limitation:** Both responses were generated by the same Claude model in a self-evaluation context. The OpenEng response had access to OpenEng skill files; the Base response did not. Human blind scoring would provide stronger evidence. The automated self-evaluation inherently cannot replicate true blinding -- the model generating the Base response is aware of the OpenEng context even when instructed not to use it. Scores should be interpreted as indicative rather than definitive.

### Cross-Domain Coverage

Prompts are selected to cover the major engineering domains in the OpenEng library, testing whether skills provide uplift across the full breadth of road engineering practice -- not just in one narrow area.

## Prompts Used

Eight prompts spanning the major domains of the OpenEng skills library:

### 1. Geometric Design

**Prompt:**
> You are designing the horizontal and vertical alignment for a 5 km section of a Class II trunk road in Kasese District, Western Uganda. The terrain is rolling with ground slopes averaging 8-15%. Design speed is 80 km/h per MoWT standards. The route must cross a river valley with a 35 m elevation difference over 800 m horizontal distance. Provide the design parameters (minimum radius, maximum gradient, K-values for vertical curves), identify the critical constraints, and propose an alignment strategy that balances earthworks cost against design standards compliance.

**Skills expected to contribute:** #13 (Design Standards Selection), #14 (Horizontal Alignment), #15 (Vertical Alignment), #16 (Sight Distance), #17 (Cross-Section Design)

### 2. Pavement Design

**Prompt:**

### 3. Drainage

**Prompt:**

The 25-year design storm intensity is 120 mm/hr. The road embankment height is 2.5 m, the natural stream bed gradient is 1.5%, and the downstream channel can accommodate up to 2.8 m3/s without flooding. **Prompt:**

### 5. FIDIC Contracts

## Analysis

### Overall OpenEng Advantage

OpenEng (skills-loaded) scored an average of +6.9 points higher than Base Claude across all 8 prompts on the 20-point scale. This represents a 56% improvement over the Base mean (12.3 to 19.1). OpenEng won all 8 prompts -- no ties and no Base wins. The advantage is consistent across domains rather than concentrated in one area, though the magnitude varies from +4 (FIDIC) to +10 (Pavement Design, Asset Management).

### Strongest Domain Uplift

**Pavement Design (+10) and Asset Management (+10)** showed the largest skill uplift. These domains share a common characteristic: they are heavily table-driven, requiring specific numeric lookup values from the MoWT standards.

For Pavement Design, OpenEng had direct access to the MoWT Vol 3 Part I catalogue tables -- traffic class boundaries (T1-T8), subgrade class boundaries (S1-S6), the wet/dry determination matrix (Table 4.1), chart classification (Table 5.3), and the design catalogue structures. Base Claude could describe the general catalogue method but lacked the exact class boundaries and table values, producing vague or approximate classifications.

For Asset Management, OpenEng had IRI thresholds (paved: Good <4, Fair 4-6, Poor 6-8, Very Poor >8 m/km), VCI condition categories, MoWT maintenance trigger levels (routine/periodic/rehabilitation thresholds for IRI, VCI, rutting, cracking extent), and the prioritization framework. Base Claude could discuss asset management concepts but could not provide Uganda-specific trigger levels or condition classification thresholds.

### Weakest Domain Uplift

**FIDIC Contracts (+4)** showed the smallest skill uplift. This is expected: FIDIC contract administration is a well-documented international practice, and Base Claude has strong training-data coverage of FIDIC 1999 Red Book clauses. Base Claude correctly identified the 28-day notice bar (Sub-Clause 20.1), the claim procedure, and the Engineer's determination role. The skill uplift came from Uganda-specific contract administration context (UNRA Particular Conditions, World Bank prior review/no-objection overlay) and the precise linking of Sub-Clause 4.12 (unforeseeable physical conditions) to the entitlement framework -- but the baseline performance was already solid.

**Drainage (+5)** also showed relatively modest uplift because the Rational Method and culvert hydraulic design are well-known engineering methods. Base Claude correctly applied Q = CIA/360, identified the need to check both inlet and outlet control, and produced reasonable calculations. The skill advantage came from specific MoWT references (Vol 2 Eqn 4.10, Section 9.10 minimum culvert size), the precise Ke values, and the HW/D ratio limits per MoWT guidance.

### Per-Criterion Analysis

**Technical Accuracy (OpenEng 4.75, Base 3.63, Delta +1.13):**
Skills provide moderate accuracy improvement. Base Claude generally applies correct engineering methods and formulas -- the accuracy gap comes from specific numeric values (table lookups, threshold values, load factors) rather than fundamental methodology errors. In domains where the method is well-known (Rational Method, FIDIC procedures), Base Claude scores 4/5 on accuracy. In table-driven domains (pavement catalogue, asset management triggers), Base accuracy drops to 3/5 due to approximated or incorrect parameter values.

**Completeness (OpenEng 4.75, Base 3.50, Delta +1.25):**
Skills improve completeness moderately. The skill files follow structured decision logic (Step 1, Step 2, etc.) which ensures all required checks are performed. For example, in pavement design, OpenEng follows the complete sequence: traffic class -> subgrade class -> wet/dry check -> chart type selection -> catalogue lookup -> practical checks. Base Claude tends to skip intermediate steps (notably the wet/dry determination in pavement design) or omit secondary considerations (climbing lane assessment in geometric design, sensitivity checks near class boundaries).

**Uganda/East Africa Relevance (OpenEng 4.88, Base 2.25, Delta +2.63):**
This is the strongest uplift criterion, as expected. Skills encode the specific MoWT table numbers, UNRA practices, Uganda institutional context (UNRA vs District roles, Uganda Road Fund, PPDA procurement), local parameters (seismic zones, climate factors, regional cost adjustment factors), and regulatory framework (NEMA, World Bank ESF). Base Claude provides generic or international-standard responses with limited Uganda specificity. In several cases, Base Claude referenced AASHTO or generic international practice where the prompt specifically asked for MoWT standards. The gap is largest for domains like pavement design and asset management where Uganda has its own classification system distinct from international practice.

**Standard Compliance (OpenEng 4.75, Base 2.88, Delta +1.88):**
Skills significantly improve standard citation accuracy. OpenEng cites specific MoWT section numbers (Vol 1 Section 6.2, Vol 3 Part I Table 2.5, Vol 4 Table 1), table references (Table 6-11, Table 4.1), and equation numbers (Eqn 4.10). Base Claude typically refers to standards generically ("per MoWT guidelines" or "according to Uganda road design standards") without specific section or table citations, or occasionally cites AASHTO/BS references that are technically correct in the international context but are not the primary Uganda standard.

### Key Observations

1. **The uplift is real but must be interpreted with caution.** Both responses come from the same model -- the OpenEng response benefits from having specific numeric data loaded into context, which is a genuine advantage but not the same as improved reasoning ability. The skill files provide authoritative reference data that the model can cite; without them, the model often produces plausible but unverifiable or approximate values.

2. **Base Claude performs better than expected on methodology.** General engineering methods (Rational Method, AASHTO pavement design concepts, FIDIC contract procedures, bridge loading principles) are well-represented in training data. The fundamental engineering reasoning is sound even without skills.

3. **The Uganda-specific gap is the clearest value proposition.** MoWT table values, UNRA institutional context, and Uganda-specific parameters are unlikely to be reliably recalled from training data. This is precisely the type of knowledge that a curated skill library should encode.

4. **Table-driven domains show the largest uplift.** Domains that require specific numeric lookup values (pavement catalogue classes, IRI thresholds, load combination factors, gradient limits by terrain class) benefit most from skills. Domains that are more reasoning-driven (FIDIC claims assessment, economic screening) show smaller but still positive uplift.

5. **The scoring methodology has inherent bias.** The same model that generated both responses also scored them. While the scoring rubric is explicit and the criteria are defined, a human evaluator might score differently -- particularly on the "Completeness" criterion where what counts as "complete" is somewhat subjective.

## Conclusions

### Evidence Summary

The A/B test provides evidence that the OpenEng skills library produces measurable uplift in response quality across all 8 cross-domain engineering prompts tested. The average improvement of +6.9 points on a 20-point scale (+56% over base) is consistent and positive across all domains and all four scoring criteria. The uplift is strongest for Uganda/East Africa Relevance (+2.63 per prompt average), which aligns with the primary design goal of the skills library: encoding authoritative local engineering knowledge that cannot be reliably sourced from model training data alone.

However, the evidence has important limitations:

1. **Self-evaluation bias:** Both generation and scoring were performed by the same model. The model generating the OpenEng response has access to authoritative skill data, giving it a structural advantage -- but the scoring model may also be biased toward responses that match the skill file content it has seen.

2. **No true blinding:** The model is aware of the OpenEng project context throughout, making it impossible to generate a truly "naive" Base Claude response.

3. **Sample size:** Eight prompts across 8 domains provides indicative coverage but not statistical significance. Two or more prompts per domain would strengthen the evidence.

4. **Single evaluator:** Automated self-evaluation. Human blind scoring by a qualified Uganda-based road engineer would provide substantially stronger evidence.

### Limitations

- Sample size: 8 prompts across 8 domains means some domains share prompts or are not directly tested
- Single evaluator: automated self-evaluation reduces but does not eliminate bias; human blind scoring would be stronger evidence
- Prompt design: prompts are realistic but pre-selected; a random sample from real project work would be stronger evidence
- Model version: results are specific to the Claude model version used; performance may vary with model updates
- Self-evaluation context: both responses generated by the same model instance -- the OpenEng response had access to OpenEng skill files; the Base response did not. This structural advantage is genuine (it reflects how the skills would be used in practice) but the scoring cannot be considered independent

### Recommendations

1. **Proceed with skills library deployment.** The evidence supports the hypothesis that skill-loaded responses are more accurate, complete, locally relevant, and standards-compliant than unloaded responses, particularly for Uganda-specific engineering knowledge.

2. **Commission human blind evaluation.** Engage 2-3 qualified East African road engineers to score a subset of response pairs (ideally all 8) under true blinding conditions. This would validate or refine the automated scores.

3. **Expand prompt coverage.** Add 2-3 additional prompts per domain to improve statistical robustness. Include edge cases (missing data, conflicting requirements, unusual site conditions) to test skill behaviour under stress.

4. **Track version-over-version performance.** Re-run the A/B test when the underlying model is updated to verify that skill uplift is maintained. Model improvements may reduce the base gap (improving base performance) or increase it (if the model becomes better at leveraging loaded context).

5. **Consider domain-specific skill refinement.** The FIDIC domain showed the smallest uplift (+4), suggesting that base model knowledge is already strong. Investment in skill refinement should prioritize table-driven domains (pavement, asset management, structures) where the uplift is largest and the authoritative data is most valuable.
